# fordb M2 — Connection Manager: Design Spec

Status: Approved 2026-07-08 · Milestone: M2 · Branch: m2-connection-manager
Supersedes the M1 "one PostgresAdapter per port" model. References: docs/06-prd.md §4–5, docs/07-work-plan.md M2.

## Goal / exit criterion

Create and save connection profiles and connect to **both a local and an SSH-tunneled remote Postgres** from the UI, landing on a live schema tree. Connection secrets are stored in the OS keychain, never persisted in plaintext.

## Non-goals (deferred to M3+)

SQL editor, results grid, query tabs, row editing, table-data browse. The M2 schema tree is a read-only proof-of-connection surface, not the full M3 browsing experience.

## 1. db-host architecture (folds in M1 carry-forward)

Replaces M1's "one `PostgresAdapter` per renderer port" with a host control layer addressed by connection id.

### ConnectionRegistry (db-host)

- State: `Map<connectionId, RegistryEntry>` where `RegistryEntry = { adapter: DbAdapter; profile: ConnectionProfile; tunnel?: TunnelHandle }`.
- Methods:
  - `openConnection(profile, secrets) → Promise<connectionId>` — establishes SSH tunnel if configured, constructs a `PostgresAdapter`, connects it, stores the entry, returns a fresh id.
  - `closeConnection(id) → Promise<void>` — disconnects the adapter (closes cursors + pg client), tears down the tunnel, deletes the entry. Idempotent.
  - `getAdapter(id) → DbAdapter` — throws if unknown id.
  - `closeAll() → Promise<void>` — for port-close teardown.
- `connectionId` is an opaque string generated by an injected id source (not `Math.random`/`Date` in workflow-forbidden contexts — use a counter + process-unique prefix).

### HostApi (the RPC target)

The renderer talks to a single control port (existing `getDbHostPort`) whose target is one `HostApi` per port, backed by one `ConnectionRegistry`:

- `testConnection(profile, secrets) → Promise<TestResult>` — open, `SELECT 1`, capture ok/error, always close. Never registers a live connection.
- `openConnection(profile, secrets) → Promise<connectionId>`
- `closeConnection(connectionId) → Promise<void>`
- `listDatabases(connectionId)`, `listSchemas(connectionId)`, `listTables(connectionId, schema)`, `getColumns(connectionId, schema, table)`, … — every `DbAdapter` introspection method, prefixed with `connectionId`, routed via `registry.getAdapter(id)`.
- This retires the M1 multi-tab correlation debt: connections are addressed by id, not by port. `getDbHostPort` still returns one `PortLike`; multiplexing is by `connectionId` in the payload, not multiple ports.

### utilityProcess supervision (main)

- `dbHost.on('exit', …)` → respawn the db-host utilityProcess and reject any in-flight `getDbHostPort` invoke so the renderer surfaces an error instead of hanging (closes M1 carry-forward item).
- On respawn, the renderer's existing connections are gone; renderer must re-open. M2 surfaces this as a "connection lost" state, not silent.

### Secrets flow

- `testConnection`/`openConnection` receive `secrets` (password, ssh passphrase) decrypted by the **main** process from the keychain and passed to db-host over IPC at call time. Secrets are never persisted in db-host and never returned to the renderer.

## 2. Persistence (main process)

- **`profiles.json`** under Electron `userData`: `ConnectionProfile[]` — host/port/db/user, SSL options (CA/cert/key paths or inline, trust toggle), SSH options (host/user/port/auth-method, key path). **No secrets.**
- **`secrets.bin`** under `userData`: password + SSH key passphrase per profile id, encrypted via Electron `safeStorage` (OS keychain). Written on profile save, read on connect.
- Main owns both files (keychain access lives in main). Renderer requests the profile list and saves profiles via IPC (`profiles:list`, `profiles:save`, `profiles:delete`). Secrets travel renderer→main on save, main→db-host on connect; they never round-trip back to the renderer.
- `ConnectionProfile.id` is the join key across both files.

## 3. SSH tunnel (db-host)

- `tunnel-ssh` wrapper: `createTunnel()` → local listener → pg connects to `127.0.0.1:<localPort>`.
- Auth methods: password, private key + passphrase, SSH agent (mirrors sqlectron's model).
- Tunnel handle owned by the `RegistryEntry`; torn down in `closeConnection`. One tunnel per connection.

## 4. UI (renderer — React 19 + Tailwind v4 + Zustand)

- **Tailwind v4** via its Vite plugin, compiled into the renderer bundle at build (no runtime fetch). First styling foundation; all later UI inherits it.
- **Zustand** store for connection state (profiles, active connections, current selection).
- **Connection list** (sidebar): saved profiles; connect / edit / delete; keyboard-navigable.
- **Profile form**: host/port/db/user/password; SSL section (CA, client cert/key, trust-server toggle); SSH section (host/user/port, auth method, key). **Test-connection** button → `HostApi.testConnection`, shows ok/error inline.
- **Connect** → keep-alive → land on **schema tree** (react-arborist, pulled forward from M3): databases → schemas → tables/views, lazy-loaded via `HostApi` introspection routed by `connectionId`. Read-only. This is M2's proof-of-life.
- **Command palette** (Ctrl+K, PRD principle): register "New connection", "Connect to…", "Disconnect" commands. Every action keyboard-reachable.

## 5. Testing

- Contract suite unchanged — still 11/11 against Dockerized Postgres.
- **ConnectionRegistry** unit tests: open returns distinct ids; close disconnects + removes; getAdapter throws on unknown id; closeAll; isolation between two connections.
- **HostApi routing** tests over the real RPC layer (node MessageChannel): a `connectionId`-prefixed call reaches the right adapter; unknown id rejects with a structured error.
- **Profile store**: profiles.json round-trip; **safeStorage** secret round-trip (mocked in unit — safeStorage unavailable headless; real in a documented smoke).
- **SSH tunnel**: integration test against a Dockerized `sshd → postgres` compose if feasible in CI; otherwise a documented manual verification + a unit test of tunnel-config construction.
- **Playwright smoke**: create profile → test-connection → connect → schema tree renders.

## 6. Success criteria

1. Save a profile; reopen app; profile persists, secret decrypts from keychain on connect.
2. Connect to local Postgres from the UI → schema tree shows the fixture's `app` schema.
3. Connect to an SSH-tunneled remote Postgres from the UI (tunnel established, torn down on disconnect).
4. Two simultaneous connections are isolated (introspection on one doesn't touch the other).
5. Killing the db-host process surfaces a "connection lost" state in the renderer, not a hang; reconnect works.
6. Full create→test→connect flow is keyboard-only reachable.

## 7. Risks

| Risk                                       | Mitigation                                                                                                                                                         |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| safeStorage unavailable in headless CI/dev | Unit tests mock it; real keychain exercised only in a documented desktop smoke; app degrades to an explicit "keychain unavailable" error, never plaintext fallback |
| SSH tunnel test infra flaky in CI          | Dockerized sshd→postgres if stable; else tunnel-config unit test + manual smoke, clearly logged as not-in-CI                                                       |
| Pulling schema tree forward bloats M2      | Tree is read-only, lazy, react-arborist only — no editing/query; capped scope                                                                                      |
| HostApi becomes a god-object               | Registry holds state; HostApi is a thin routing facade; introspection methods delegate to `getAdapter(id)`                                                         |

## Decisions made during design (not asked)

- **RPC shape**: single control port + `connectionId`-addressed `HostApi`, not one-port-per-connection — cleaner, retires multi-tab correlation debt.
- **Schema tree pulled forward** into M2 as the connect payoff.
- **Tailwind v4** adopted now (user request) as the styling foundation.
